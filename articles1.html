How to Maintain Clean Code

February 26th 2022

By Eric Liang

One of the biggest nuisances of programming doesn’t come from actually coding, but rather maintaining your code. Something as small as letting your code go messy can have lots of consequences such as making it difficult to understand, difficult to add onto the code, and even result in errors. Thus, if you ask any experienced programmer, they’ll tell you the importance of learning how to maintain code.

Tips for Tidying Code

Regardless of whether you are in desperate need of fixing your messy code or just want a quick refresher on how to improve your programs, there are a few ways to prune over previous code to make it look better.

First of all, it’s very important to use representative and descriptive variable names. This helps ensure that people can understand what’s happening in the code, even if you don’t include any documentation. Ideally, the variable names should be understandable to somebody who is reading the code for the first time without knowledge of what the code does, a general trend that occurs for most programming practices. As a result, it’s not a good idea to use abbreviations in your code:

n = a + b

What does a + b mean? Apples + bananas? Abbie + Ben? It makes much more sense to actually describe what the variables represent. Generally, don’t worry too much about the length of the variable as long as it isn’t redundant and adequately describes the code.

Documentation is another important part of code since it explains what the code does; this helps any future programmers and potentially a future you whenever you’re reading back your code. Documentation can be kept in comments and should generally be used to describe what the code does in broad strokes. It would be redundant to use a comment for each tiny detail in the code, as shown below:

print(x + y) #Finds the sum of x + y and prints it out

This is a bit of an exaggeration but this type of documentation is clearly still quite silly. In general, it’s recommended to use documentation to explain larger things that someone might have difficulty understanding, such as algorithms or functions. It’s also important to keep the documentation simple and concise, as shown below:

# Dijkstra’s algorithm by repeatedly finding the nearest unvisited node
def dijkstra(self, src):

    dist = [1e7] * self.V
    dist[src] = 0
    sptSet = [False] * self.V

    for cout in range(self.V):

        # Pick the minimum distance vertex from
        # the set of vertices not yet processed.
        u = self.minDistance(dist, sptSet)

        sptSet[u] = True

        # Update dist value of the adjacent vertices
        for v in range(self.V):
            if (self.graph[u][v] > 0 and
                sptSet[v] == False and
                dist[v] > dist[u] + self.graph[u][v]):
                dist[v] = dist[u] + self.graph[u][v]

    self.printSolution(dist)

Lastly, a good programming practice is to limit the functions to a single task to execute and name it accordingly, rather than doing multiple things at once. This makes the code much more understandable and also easier to use. For instance, if the function findSquare() returns the area, coordinates, and side length, then the function is quite confusing for other programmers. As a result, it’s much better to split the function into findSquareArea(), findSquareLength(), and findSquareCoordinates(), eliminating the confusion.

Good Programming Practices

The things above make your code far more readable and functional, but there are also other tips that can greatly improve the quality of someone’s code. Otherwise, your code may be considered “spaghetti code,” which stands for messy and unstructured code.

Code is often condensed and stretched hundreds if not thousands of lines, so it’s important to partition your code into sections that are logically consistent. For example, if you should probably have a line break between getting input and processing the input, since it makes much more sense that they’re separate actions. However, it’s also important to use line breaks inside each section to make the code even more readable, such as separating for loops and declaring variables.

There are also other common code practices such as naming conventions for variables. For instance, camel case separates words in variables by uppercasing them and snake case uses an underscore. Another heated debate is whether you use tabs or spaces to indent your code. Generally, there are pros and cons for both sides of these arguments, so it ultimately depends on personal preference. However, it’s quite important to have consistency throughout the entire code to make it visually pleasing and less confusing.

Clean Code and its Benefits

As mentioned above, clean and consistent code is incredibly important because it essentially makes the code much easier to read. Since code often goes for long periods of time before being updated, clean code ensures that the next person to change the code, whether a different programmer or yourself, won’t have difficulty deciphering the code, helping save lots of time.

Clean code also reduces the amount of errors since it helps straighten out your thoughts so you know exactly what it does. Any misunderstanding errors that might arise from bad code won’t happen, which can reduce the amount of errors.

Through these tips, you can also write clean code. Just remember, no matter how much you enjoy Italian food, don’t get tempted by the spaghetti code!
